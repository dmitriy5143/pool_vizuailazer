# План тестирования image-моделей

Цель: выбрать модель/провайдера, который лучше всего подходит для демо AI-визуализатора бассейнов.

## 0. Покрытие текущего демо

В текущую версию демо входит:

- загрузка фото участка;
- выделение прямоугольной зоны размещения бассейна на фото;
- заполнение параметров бассейна: размер, форма, стиль, материалы, комментарий;
- генерация 3-5 вариантов через OpenRouter Image API;
- mock-режим без OpenRouter-ключа для проверки интерфейса;
- параллельная генерация вариантов через отдельные `n=1` вызовы;
- capability-aware OpenRouter adapter: в запрос отправляются только параметры, которые поддерживает выбранная модель;
- локальное сохранение загруженных фото и сгенерированных картинок;
- история генераций в file-based demo-DB `data/requests.json`;
- галерея результатов;
- выбор лучшего варианта менеджером;
- scorecard для оценки вариантов;
- экспорт CSV/JSON;
- Docker-запуск с volume для `data/`;
- тестовый набор из 20 фото с зонами и параметрами.

Легкий технический pre-check уже есть: обязательные поля, размер зоны, допустимые значения параметров, формат/размер файла, предупреждение о низком разрешении. Полноценный automatic pre-check фото/зоны как отдельный AI/vision-этап пока не входит в упрощенное демо.

За рамками текущего демо и может быть добавлено позже:

- брендированная клиентская страница proposal-lite;
- photo guide перед загрузкой фото;
- polygon/brush-mask вместо прямоугольника;
- полноценная БД с проектами, клиентами и авторизацией;
- автоматическая серверная batch-оценка моделей без ручного UI-прогона.

Проверяем не просто красоту картинки, а пригодность результата для продаж:

- модель сохраняет исходный участок;
- бассейн попадает в выделенную зону;
- не ломаются дом, забор, деревья и перспектива;
- вода, материалы и тени выглядят правдоподобно;
- результат можно показать клиенту как концепт.

## 1. Подготовка фото

Стартовый набор уже собран: 20 публичных фото из Wikimedia Commons с лицензиями и атрибуцией.

Набор покрывает:

- чистый daylight baseline;
- дом рядом с зоной;
- узкую перспективную зону;
- маленькую patio-зону;
- естественный сад с деревьями;
- wide/premium lawn;
- мебель и препятствия;
- склон/сложную перспективу;
- забор рядом с зоной;
- среднее качество фото;
- hardscape/patio;
- rough construction-like plot;
- более клиентские сценарии TC-13...TC-20: suburban backyard, townhouse patio, brick townhouse yard, hardscape backyard, rear yard, patio with furniture.

Файлы:

```text
public/test-photos/
public/test-cases.json
docs/model-test-cases.csv
docs/test-photos/sources/test-cases.csv
```

Быстрый просмотр набора с зонами: `http://localhost:5177/test-photo-gallery.html`.

## 2. Настройка OpenRouter

В `.env`:

```env
DEMO_GENERATION_MODE=openrouter
OPENROUTER_API_KEY=sk-or-...
OPENROUTER_IMAGE_MODEL=...
```

Важно: модель должна поддерживать image input / reference image. Если модель не поддерживает image input, приложение вернет понятную ошибку.

## 3. Методика теста

Тестируем экономно: не прогоняем все модели по всему набору. Сначала выбираем лидера на малом числе кейсов, затем глубже исследуем только его. Если лидер провалится на расширенном наборе, переходим к следующей модели из короткого списка.

### Этап 1. Быстрый отсев моделей

Кандидаты:

- `openai/gpt-image-2`;
- `openai/gpt-image-1-mini`;
- `bytedance-seed/seedream-4.5`;
- `google/gemini-3.1-flash-image`;
- опционально `black-forest-labs/flux.2-pro`, если первые модели не дают нужный фотореализм.

Стартовые кейсы:

- `TC-13` — типовой клиентский backyard;
- `TC-14` — маленький townhouse patio;
- `TC-17` — low-res/hardscape клиентский стресс-кейс.

Объем:

```text
4 модели x 3 кейса x 3 варианта = 36 изображений
```

Цель этапа: быстро отсеять модели, которые плохо сохраняют участок, не попадают в зону или дают слишком много AI-артефактов.

### Этап 2. Углубление по лучшей модели

Берем лучшую модель с этапа 1 и прогоняем больше кейсов:

```text
1 модель x 8-10 кейсов x 3-5 вариантов = 24-50 изображений
```

Рекомендуемые дополнительные кейсы:

- `TC-03` — узкая перспектива;
- `TC-09` — забор рядом с зоной;
- `TC-15` — brick townhouse small yard;
- `TC-16` — terrace/building edge;
- `TC-20` — patio with furniture.

Цель этапа: понять, можно ли модель брать как основную для демо, а не просто как удачный результат на легких фото.

### Этап 3. Переход к следующей модели при провале

Если лучшая модель на этапе 2 не дает стабильного результата, не расширяем ее дальше. Берем следующую модель из рейтинга этапа 1 и прогоняем тот же расширенный набор.

### Этап 4. Финальная проверка перед демо

Когда основная модель выбрана:

```text
1 основная модель x 20 кейсов x 3 варианта = 60 изображений
```

Этот этап нужен только перед финальным демо или если нужно доказательно показать устойчивость решения.

Ожидаемый бюджет на первичный выбор модели: до `$10`, если не делать много повторных генераций на дорогих настройках.

Для каждого запуска:

1. Запустить приложение.
2. В блоке `Фото и зона` выбрать тестовый кейс из набора.
3. Нажать `Загрузить`: демо подставит фото, зону, параметры и `case_id`.
4. Генерировать 3-5 вариантов.
5. Оценить варианты прямо в scorecard интерфейса.
6. Экспортировать CSV/JSON результата.

## 4. Оценка результата

Каждую генерацию оцениваем по шкале 1-5:

| Критерий | 1 | 3 | 5 |
|---|---|---|---|
| Сохранение участка | Участок сильно изменен | Есть заметные изменения | Дом/забор/ракурс сохранены |
| Попадание в зону | Бассейн не там | Примерно в зоне | Внутри выделенной зоны |
| Реалистичность | Похоже на AI-артефакт | Приемлемо | Выглядит правдоподобно |
| Соответствие параметрам | Не похоже | Частично похоже | Форма/стиль/материалы совпадают |
| Артефакты | Много грубых ошибок | Есть мелкие ошибки | Грубых ошибок нет |
| Можно показать клиенту | Нет | Только после отбора | Да |

## 5. Критерий успеха

Модель подходит для демо, если:

- не менее 60-70% результатов можно показать как концепт после одной генерации или одной перегенерации;
- средняя оценка `sendable` выше у одной модели по сравнению с альтернативами;
- модель стабильно сохраняет участок и не меняет ключевые объекты вне зоны;
- стоимость одной пригодной картинки приемлема для пилота.

## 6. Что сравниваем между моделями

- качество;
- стабильность;
- скорость;
- стоимость;
- поддержка image input / mask reference;
- насколько часто нужен повторный запуск.

## 7. Решение после теста

После теста выбираем:

- основную модель для демо;
- fallback-модель, если основная дает плохой результат или ошибку;
- правила для фото, которые нельзя отправлять в генерацию без улучшения;
- нужно ли оставлять прямоугольник или делать polygon/brush-mask.
