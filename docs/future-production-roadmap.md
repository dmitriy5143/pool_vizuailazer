# Future Production Roadmap

## Цель

Текущее демо проверяет базовую гипотезу: пользователь загружает фото участка, задает видимый контур бассейна, параметры и получает 3-5 AI-вариантов через внешний image generation API.

Более сильное production-решение должно работать иначе: генеративная модель не должна сама решать, где находится бассейн и какого он размера. Эти решения должен принимать инженерный слой системы, а модель должна только фотореалистично вписывать уже рассчитанный объект в исходную фотографию.

Ключевой принцип:

```text
Геометрия и ограничения считаются детерминированно.
AI используется для визуализации, сегментации, проверки и фотореалистичного дорисовывания.
```

## Почему одного prompt + mask недостаточно

Image editing модель может получить корректную маску, размеры и текстовые ограничения, но она все равно интерпретирует сцену вероятностно. Поэтому возможны ошибки:

- бассейн оказывается на заборе, стене или фасаде;
- модель меняет исходный двор вместо локального редактирования;
- размер выглядит визуально неверным;
- бассейн попадает примерно в область, но не лежит на плоскости земли;
- появляются лишние объекты, текст, подписи, мебель, декор;
- перспектива воды, бортика и плитки не совпадает с фото.

Для демо это допустимо, если есть pre-check и ручной отбор. Для production такого уровня контроля мало.

## Целевая архитектура

Production-уровень должен состоять из нескольких слоев.

```text
Фото участка
  -> Scene understanding
  -> Ground plane / scale calibration
  -> CAD / 2.5D / 3D placement
  -> Control layers
  -> Controlled image generation
  -> Multi-stage validation
  -> Manager review / proposal
```

## 1. Scene Understanding

На этом этапе система должна понять, что находится на фото.

Нужно определить:

- землю, газон, патио, дорожки;
- дом, фасад, стены, крышу;
- заборы и ограждения;
- деревья, кусты, крупные растения;
- мебель и другие объекты;
- потенциально редактируемые зоны;
- зоны, которые нельзя менять.

Возможный стек:

- open-vocabulary object detection для поиска объектов по текстовым категориям;
- segmentation model для точных масок объектов;
- VLM для общей интерпретации сцены и рисков;
- классические CV-проверки для границ, пересечений и качества фото.

Пример логики:

```text
Найти "fence", "house", "wall", "tree", "patio", "grass".
Построить forbidden masks для дома, забора, стен, деревьев.
Построить editable ground mask для земли/газона/патио.
Запретить размещение бассейна вне editable ground mask.
```

Роль сегментатора здесь не в том, чтобы "сделать красиво", а в том, чтобы дать точные пиксельные маски сущностей.

## 2. Ground Plane And Scale

Главная проблема текущего демо: пользователь задает длину и ширину в метрах, но фото является 2D-проекцией 3D-сцены. Чтобы бассейн выглядел инженерно правдоподобно, нужно оценить плоскость земли и масштаб.

Минимальный production-подход:

- пользователь проводит одну линию известной длины;
- система считает `pixels per meter`;
- размер бассейна в метрах переводится в видимый footprint;
- пользователь видит контур, который реально уйдет в генерацию.

Более сильный подход:

- пользователь задает 2-3 калибровочные линии;
- система оценивает перспективу;
- строит локальную плоскость земли;
- переводит размеры бассейна в перспективный quadrilateral footprint;
- показывает не прямоугольник, а перспективную 2.5D-форму на земле.

Еще сильнее:

- depth estimation помогает понять относительную глубину сцены;
- vanishing point / homography estimation уточняет перспективу;
- known object scale или ручная калибровка дает метрический масштаб.

Важно: depth model сама по себе не гарантирует точные метры. Для инженерной достоверности нужна калибровка пользователем или дополнительные измерения.

## 3. CAD / 2.5D / 3D Placement Layer

Это ключевое отличие сильного решения от простого AI-demo.

CAD/3D слой должен рассчитывать:

- форму бассейна;
- реальные габариты;
- поворот;
- положение на земле;
- бортик;
- ступени;
- зону отделки;
- отступы от дома, забора, деревьев;
- коллизии с запрещенными объектами;
- видимую проекцию бассейна в пикселях.

На выходе этот слой должен давать не только одну маску, а набор control layers.

Пример:

```text
pool_footprint_mask.png
editable_area_mask.png
preserve_mask.png
forbidden_objects_mask.png
ground_plane_mask.png
depth_map.png
edge_map.png
rough_3d_render.png
```

Именно этот слой отвечает за инженерную часть. Модель генерации не должна придумывать геометрию с нуля.

## 4. Control Layers For Generation

Для более надежной генерации модели нужно передавать не только текстовый prompt.

Нужно передавать:

- исходное фото;
- точную edit mask;
- footprint будущего бассейна;
- preserve mask;
- forbidden masks;
- rough render бассейна;
- depth map;
- edge map;
- semantic map;
- параметры в structured form.

Идеальная логика:

```text
Модель не выбирает место бассейна.
Модель не выбирает размер бассейна.
Модель не меняет камеру.
Модель не меняет дом, забор и деревья.
Модель только фотореалистично вписывает рассчитанный объект в заданные пиксели.
```

Для моделей, которые поддерживают spatial conditioning, можно использовать подходы уровня ControlNet / depth control / segmentation control. Если внешний API не дает такого контроля, надо выбирать провайдера, который поддерживает более строгий image editing / inpainting / reference conditioning.

## 5. Generation Pipeline

Целевой pipeline:

```text
1. User uploads photo.
2. System runs photo quality check.
3. System detects scene objects.
4. User confirms or edits ground area.
5. User calibrates scale.
6. User sets pool parameters.
7. CAD/2.5D layer computes footprint.
8. System checks collisions and placement constraints.
9. System builds control layers.
10. Image model generates variants.
11. Validator checks every variant.
12. Manager sees only show/review-safe results.
13. Hidden variants are available only for internal troubleshooting.
```

## 6. Stronger Validation Gate

Production validation should not rely only on one VLM answer.

Нужно несколько проверок:

### Deterministic Mask Checks

- измененные пиксели в основном находятся внутри edit mask;
- generated pool segmentation пересекается с planned footprint;
- бассейн не пересекает forbidden masks;
- зона дома, забора, деревьев не изменилась сверх порога.

### Segmentation Checks

- повторно сегментировать сгенерированный бассейн;
- сравнить фактическую маску бассейна с planned footprint;
- проверить, что бассейн лежит на ground mask;
- проверить, что вода/бортик не попали на забор или стену.

### Depth / Perspective Checks

- проверить, что footprint расположен на плоскости земли;
- проверить, что перспектива бортика и воды согласована с фото;
- проверить, что объект не "висит" и не уходит в вертикальные поверхности.

### VLM Checks

- оценить сохранение участка;
- оценить попадание в зону;
- оценить реалистичность;
- оценить соответствие параметрам;
- оценить артефакты;
- отдельно спросить: "Можно ли показать клиенту?"

### Policy

```text
show   -> можно показывать менеджеру как хороший вариант
review -> менеджер должен проверить
hide   -> не показывать в обычной галерее, только внутренний просмотр
```

Для production лучше делать conservative gate:

- явные галлюцинации всегда `hide`;
- сомнительные варианты всегда `review`;
- `show` только при прохождении всех критичных проверок.

## 7. User Experience

Для пользователя сильное решение должно выглядеть просто, хотя внутри оно сложное.

Первый экран:

- загрузка фото;
- подсказки по качеству фото;
- автоматическое определение участка;
- простая коррекция зоны;
- калибровка масштаба;
- параметры бассейна;
- запуск генерации.

Пользователь должен видеть:

- где именно будет бассейн;
- как длина и ширина влияют на контур;
- какие объекты система считает запрещенными;
- почему вариант скрыт или отправлен на ревью;
- что исправить для следующей генерации.

Не нужно показывать пользователю все технические маски по умолчанию. Но для менеджера/админа полезен debug mode:

- original photo;
- user zone;
- ground mask;
- forbidden mask;
- generated footprint;
- final image;
- validation report.

## 8. Роль текущего демо

Текущее демо уже закрывает первый уровень:

- загрузка фото;
- тестовые кейсы;
- ручная зона;
- калибровка масштаба;
- параметры бассейна;
- OpenRouter image generation;
- очередь задач;
- история;
- pre-check;
- scorecard;
- hidden/review/show статусы;
- feedback для перегенерации.

Это хороший MVP для проверки коммерческой гипотезы и выбора модели.

Но текущее демо пока не делает:

- автоматическую сегментацию сущностей;
- ground plane estimation;
- перспективный quadrilateral footprint;
- CAD/3D placement;
- deterministic overlap validation;
- depth-based validation;
- ControlNet-like conditioning.

### Позиционирование развития для заказчика

Текущее демо следует позиционировать как первый рабочий уровень управляемой AI-визуализации, а не как финальную инженерную систему.

Рекомендуемая формулировка:

> Сервис уже создает визуализацию по фотографии, зоне размещения и параметрам выбранной чаши без ручного составления сложного промпта. Следующий этап включает более точную калибровку масштаба по требованиям и исходным данным заказчика, строгую разметку сцены и независимую проверку LLM-as-a-Judge. Это повысит сохранение размеров, участка и геометрии чаши, а сомнительные результаты будут автоматически скрываться до ручной проверки.

Ключевые обещания будущего решения:

- глубокая работа с масштабом, перспективой и сохранением заданных размеров;
- уточнение требуемой точности и состава входных данных вместе с заказчиком;
- простой пользовательский ввод вместо самостоятельного написания большого промпта;
- быстрое уточнение результата через параметры, оценки и короткие комментарии;
- строгая сегментация земли, дома, забора и других объектов;
- LLM-as-a-Judge как независимый слой проверки вместе с детерминированными CV-проверками;
- автоматическое скрытие явно некорректных вариантов до показа клиенту.

Нельзя обещать абсолютное отсутствие галлюцинаций: ни image model, ни VLM-validator не дают математической гарантии. Корректное обещание — последовательное снижение риска, многоступенчатая проверка и недопуск сомнительных изображений в клиентский показ.

Точность размеров также зависит от входных данных. Одно фото дает приближенную геометрию; известные расстояния, замеры участка, план или опорные точки позволяют перейти к значительно более точной калибровке.

## 9. Рекомендуемый Roadmap

### Этап 1. Усилить текущий MVP без тяжелого ML

Цель: убрать основные UX-риски и сделать демо убедительнее.

- улучшить калибровку масштаба;
- добавить перспективный четырехугольник вместо простого прямоугольника;
- добавить ручное вращение/наклон footprint;
- сохранить calibration metadata в задаче;
- добавить debug view для маски и отправленных параметров;
- добавить more explicit generation prompt preview для разработчика.

### Этап 2. Scene Segmentation

Цель: система начинает понимать объекты на фото.

- добавить сегментатор для ground/house/fence/tree/furniture;
- дать пользователю подтвердить или поправить маски;
- построить forbidden masks;
- проверять, что бассейн не пересекает forbidden masks;
- подсвечивать рискованные зоны до генерации.

### Этап 3. Ground Plane And Perspective

Цель: бассейн выглядит как объект на земле, а не как 2D-наклейка.

- добавить 2-3 калибровочные линии;
- построить локальную homography для плоскости земли;
- конвертировать размеры бассейна в perspective footprint;
- отображать footprint как перспективный quad;
- проверять footprint относительно ground mask.

### Этап 4. 2.5D / CAD Placement

Цель: параметры бассейна становятся инженерной моделью.

- добавить внутреннюю модель бассейна;
- поддержать форму, бортик, ступени, покрытие;
- считать отступы и коллизии;
- генерировать rough render;
- отдавать rough render как reference/control.

### Этап 5. Strong Validation

Цель: минимизировать показ галлюцинаций.

- сравнивать planned footprint и actual generated pool mask;
- проверять changed pixels outside mask;
- проверять forbidden object preservation;
- запускать VLM validation;
- делать ensemble decision;
- вести validation trace для каждой картинки.

### Этап 6. Production Workflow

Цель: сделать сервис полезным менеджерам и клиентам.

- клиентская proposal-lite страница;
- роли менеджера и админа;
- архив и история проектов;
- экспорт в PDF/CRM;
- брендированные варианты;
- сохранение всех control layers;
- аналитика качества моделей;
- расчет стоимости генераций.

### Этап 7. Интерактивная 3D-сцена

Цель: после утверждения визуальной концепции дать клиенту возможность осмотреть будущий участок в интерактивном пространстве.

Предлагаемый маршрут:

```text
Исходное фото + утвержденная визуализация + референс чаши
  -> World Generation API
  -> асинхронная генерация пространственной сцены
  -> просмотр сцены в браузере
  -> при необходимости экспорт spatial representation / mesh
```

Для этого можно исследовать World Labs API, который принимает одно или несколько изображений и создает интерактивный AI-generated world. Такой этап является продолжением после выбора лучшей 2D-визуализации и не входит в ближайшее демо.

Важно правильно ограничить обещание:

- World Labs используется для презентационной пространственной сцены и эффекта осмотра;
- результат нельзя автоматически считать инженерно точным CAD-двойником участка;
- точные размеры и положение бассейна должны по-прежнему определяться geometry/CAD слоем;
- если нужно гарантированно сохранить конкретную чашу во всех ракурсах, понадобится ее точная CAD/GLB-модель или отдельный параметрический 3D-объект;
- генерация является асинхронной, поэтому интерфейс должен показывать задачу, прогресс, повторные попытки и готовый результат.

Ориентир по стоимости на июль 2026 года: multi-image draft составляет около `$0.20`, standard — около `$1.28`; Marble 1.1 Plus может стоить примерно `$1.28-2.48` в зависимости от обработки. Опциональный HQ mesh GLB export стоит около `$2.80`. Перед production-запуском цены и доступные форматы нужно проверить повторно.

Официальные источники:

- [World Labs API Pricing](https://docs.worldlabs.ai/api/pricing)
- [World Generation API](https://docs.worldlabs.ai/api/reference/worlds/generate)
- [Rendering SPZ](https://docs.worldlabs.ai/api/rendering-spz)

## 10. Техническая архитектура Production

Возможное разделение сервисов:

```text
frontend
  React / TypeScript

backend-orchestrator
  Node.js / TypeScript
  API, очередь, задачи, история, OpenRouter/provider adapters

vision-service
  Python
  segmentation, depth, homography, image analysis

geometry-service
  Python or Node.js
  CAD/2.5D placement, masks, constraints

storage
  S3-compatible object storage
  PostgreSQL for tasks and metadata

workers
  generation workers
  validation workers
  image processing workers
```

Для текущего MVP нормально, что backend написан на Node.js. Но для production CV/ML-часть лучше вынести в Python-сервис, потому что там зрелее экосистема для segmentation, depth estimation, OpenCV, PyTorch и model serving.

## 11. Главные риски

- одно фото не всегда дает точный метрический масштаб;
- depth estimation может ошибаться на reflective/transparent surfaces;
- сегментация может неверно отличить патио от стены или забора;
- внешние image models по-разному соблюдают маски;
- VLM validation не является математической гарантией;
- слишком жесткий validator может блокировать хороший пользовательский путь;
- слишком мягкий validator может пропускать галлюцинации.

Поэтому production-решение должно сочетать автоматизацию и human-in-the-loop.

## 12. Вывод

Более сильное решение должно выглядеть как инженерно управляемый image editing pipeline:

```text
AI помогает понять сцену и красиво визуализировать результат.
CAD/geometry слой отвечает за размеры, положение и ограничения.
Validator отвечает за безопасность показа и качество результата.
Менеджер остается финальным фильтром перед клиентом.
```

Такой подход существенно надежнее простого prompt engineering, потому что модель генерации перестает быть единственным источником истины по геометрии.

## Ориентиры по технологиям

- SAM / SAM 2: интерактивная сегментация объектов на изображениях и видео.
- Grounding DINO / open-vocabulary detection: поиск объектов по текстовым категориям.
- Depth Anything V2 / monocular depth estimation: оценка глубины по одному изображению.
- ControlNet-like conditioning: управление генерацией через spatial controls, например depth, edges, segmentation.
- OpenCV / homography estimation: расчет перспективы и проекции на плоскость.
- Three.js / CAD kernel / custom geometry solver: визуальное и инженерное размещение бассейна.
